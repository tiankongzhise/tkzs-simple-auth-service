# 简单鉴权与限流服务产品文档（最终开发对齐版）
## 1. 产品概述
### 1.1 产品定位
本产品是**轻量级统一鉴权+精细化限流+服务治理**一体化完整平台，原生兼容 **APISIX 官方 openid-connect 插件**，无需自定义网关插件；提供网页登录鉴权、JWT无感刷新、M2M机器鉴权双模式，内置可视化管理UI（与Go服务一体化部署，无需独立部署）。
底层采用 PostgreSQL 做持久化、Redis 做缓存与分布式限流，为中小型微服务集群提供统一身份认证、权限隔离、流量管控、服务注册发现、健康检测全套能力，大幅降低业务系统单独开发鉴权限流模块成本。

### 1.2 核心目标
- 支持网页用户鉴权、M2M机器鉴权两种模式，满足人机差异化调用场景；
- 支持 Access Token + Refresh Token 双令牌机制，实现无感刷新与自动续签；
- 原生实现 **OIDC/OAuth2.0 标准协议**，无缝对接 APISIX openid-connect 官方插件；
- 内置轻量化管理UI，集成用户、APP、角色权限、服务、限流规则、黑名单可视化管理；
- 严格资源归属权限隔离，超级用户全局管控、普通用户仅管理自有资源；
- 基于**Redis令牌桶算法**实现多维度多时间粒度分布式限流，解决流量突刺，自带黑名单攻防能力；
- 支持服务注册、服务发现、定时健康检测，闭环服务生命周期管理；
- 数据库、缓存故障具备完善降级策略，保障高可用；
- 前后端一体化打包部署，单二进制启动，访问 `/ui/` 直接进入管理后台。

### 1.3 适用场景
适用于中小型服务集群、微服务架构、接入APISIX网关的业务场景，需要统一登录鉴权、机器间调用鉴权、接口流量限流、服务注册与健康监控的系统，可快速无感集成现有架构。

## 2. 核心功能设计
### 2.1 鉴权功能
#### 2.1.1 网页登录鉴权（用户级鉴权）
面向自然人用户网页登录，采用 **JWT + Redis 缓存** 实现：
1. 用户在UI输入用户名、密码发起登录；
2. 服务优先查Redis密码缓存，未命中则查询PostgreSQL；
3. 密码采用 **bcrypt 加盐哈希校验**，废弃原MD5方案；
4. 校验通过生成JWT令牌，载荷包含用户ID、角色、过期时间，同时写入Redis缓存；
5. 业务请求Header携带 `Authorization: Bearer {token}` 完成鉴权；
6. 支持令牌自动过期、手动登出销毁缓存令牌；
7. 支持 Access Token + Refresh Token 双令牌机制，Access Token 临近过期时服务端自动续签，并通过响应头返回新令牌；Access Token 已过期但 Refresh Token 有效时，客户端可调用刷新接口完成无感续签；
8. 安全策略：**连续5次密码错误锁定账号15分钟**，超时自动解锁，管理员可手动解锁。

#### 2.1.2 M2M鉴权（机器级鉴权）
面向服务间无感知调用，采用 **APPID + APPSecret + 签名校验**：
1. 用户在UI创建APP，系统自动生成：
   - APPID：8位字母数字组合，唯一不可修改；
   - APPSecret：16位字母/数字/特殊符号，**仅创建时可见，加密存储**，支持重置；
2. 下游服务请求头固定携带：`appId`、`timestamp`（秒级时间戳）、`sign` 签名；
3. 签名统一规则：
   - 请求参数按ASCII码升序排序；
   - 拼接格式：`APPSecret&timestamp&k1=v1&k2=v2`；
   - 采用 **HMAC-SHA256** 生成小写十六进制签名；
4. 服务端校验：APPID有效性、时间戳±30s防重放、签名防篡改；
5. 校验通过放行，失败返回对应错误码（APP无效、签名错误、请求超时）。

#### 2.1.3 OIDC 协议适配
服务原生实现标准 OIDC/OAuth2.0 端点，**直接适配 APISIX openid-connect 官方插件，无需自定义Lua插件**：
- 提供OIDC自动发现、授权、令牌发放、用户信息、JWKS公钥验签全套标准接口；
- 令牌采用 **RSA256 非对称加密签名**，APISIX可自动验签、信任鉴权结果；
- 提供独立限流校验接口，供APISIX网关联动做二次流量管控。

### 2.2 可视化UI功能（轻量化简化版）
UI 静态资源内嵌至Go服务，**无需独立部署、无需Nginx**，部署Go服务后直接访问 `/ui/` 进入后台；采用极简设计，剔除复杂冗余功能，仅保留核心管理能力，所有操作遵循权限隔离规则。

#### 2.2.1 用户管理
- 用户注册：普通用户自主注册，用户名3-20位（字母/数字/下划线），密码8-20位必须包含大小写+数字+特殊符号；
- 用户编辑：普通用户仅可修改自身信息；超级用户可编辑、禁用/启用所有用户；
- 账号锁定：登录连续5次失败自动锁定15分钟。

#### 2.2.2 APP管理
- APP创建：绑定创建者账号，自动生成APPID/APPSecret；
- APP编辑/删除：普通用户仅操作自有APP，超级用户全量管理；
- Secret重置：重置后旧密钥立即失效，仅创建时可查看新密钥。

#### 2.2.3 角色与权限管理
- 预设系统基础权限：用户管理、APP管理、角色权限、服务管理、限流规则、黑名单管理；
- 支持角色创建、权限分配、用户/APP与角色多对多关联；
- 权限隔离：普通用户仅能分配自身已拥有的权限，不可越权赋权。

#### 2.2.4 服务管理
- 支持UI手动创建、接口注册两种服务接入方式；
- 普通用户创建服务需超级用户审核，管理员创建直接生效；
- 支持配置服务地址、端口、自定义健康检测接口路径；
- 可为自有服务配置独立限流规则，超级用户可配置全局限流规则。

### 2.3 权限隔离功能
严格遵循**资源归属隔离原则**：
- 用户隔离：普通用户仅查看编辑自身账号，超级用户管理全量用户；
- APP隔离：仅可管理自有APP，管理员可操作所有APP；
- 角色隔离：仅可管理自建角色，不可查看他人角色；
- 服务隔离：仅可管理自有创建服务，管理员审核及全量管理；
- 限流规则隔离：仅可配置自有服务限流，管理员可配置全局及所有服务规则；
- 日志隔离：普通用户仅查看自身关联日志，管理员可查看全量日志、黑名单日志。

### 2.4 限流功能（精细化修正版）
基于 **Redis分布式令牌桶算法** 实现，替代原有固定窗口计数器，解决临界值流量突刺；Redis宕机自动降级为**本地内存令牌桶**固定阈值限流；支持多维度、多时间粒度、黑名单、白名单、告警统计能力。

#### 2.4.1 限流规则基础配置
- 支持单服务独立配置 或 继承全局默认规则；
- 配置项：限流维度、时间粒度、阈值、黑名单触发条件、封禁时长；
- 规则配置即时生效，无需重启服务。

#### 2.4.2 多时间粒度限流
支持同时启用、独立配置阈值：
- QPS秒级限流：适用于高频波动接口；
- 分钟级限流：适用于平稳流量总量管控；
- 可选扩展：小时/日级限流，满足大盘流量管控。

#### 2.4.3 多维度限流
支持单维度或多维度组合（组合取最严格阈值生效）：
- IP维度：防单IP恶意刷请求、攻击；
- 用户维度：限制单用户资源占用；
- APP维度：管控M2M单服务调用量；
- 接口维度（可选）：服务下细分接口独立限流。

#### 2.4.4 黑名单机制
基于鉴权失败滑动窗口计数（10分钟无失败自动清零）：
- 临时黑名单：连续鉴权失败达阈值，封禁默认15分钟（可配置）；
- 永久黑名单：临时黑名单累计触发达阈值自动升级；
- 权限管控：超级用户可手动移除永久黑名单，普通用户仅查看不可操作；
- 黑名单请求统一返回403，临时黑名单携带到期时间。

#### 2.4.5 限流触发与处理
- 限流触发返回标准 429 Too Many Requests，响应头携带剩余配额、冷却时间；
- 黑名单触发返回403 Forbidden，区分临时/永久封禁；
- 全量限流、黑名单、鉴权事件落日志，按权限隔离查看；
- 降级策略：Redis宕机限流自动切本地固定QPS阈值，Redis恢复自动重建缓存与规则。

#### 2.4.6 完善补充能力
原生纳入产品能力：
- 平滑限流：默认令牌桶算法，彻底解决固定窗口流量突刺；
- 白名单机制：支持IP/用户/APP白名单，不受限流和黑名单限制；
- 限流告警：连续触发限流可推送告警通知；
- 限流统计可视化：UI展示请求量、限流次数、黑名单统计，支持按时段查询；
- 弹性削峰：支持高低峰自动动态调整限流阈值。

### 2.5 服务注册与发现
#### 2.5.1 服务注册
两种接入方式：
- 接口注册：下游调用注册接口提交服务信息，需管理员审核；
- UI注册：页面手动创建，普通用户需审核、管理员直接生效；
- 超时清理：接口注册服务7天未审核自动删除。

#### 2.5.2 服务发现
- 基于Redis缓存健康服务列表，实时同步上下线、状态变更；
- 支持按服务名、健康状态筛选查询；
- 仅已审核+健康的服务可被发现调用。

### 2.6 服务健康检测
- 默认检测间隔30秒，管理员可配置10秒~5分钟；
- 检测默认路径 `/health`，**支持自定义健康接口路径**；
- 状态分级：健康 → 亚健康（单次异常）→ 异常（连续3次异常）；
- 异常服务自动移出服务发现列表，恢复后自动重新纳入；
- 异常触发告警通知，全量检测日志按权限隔离查看。

### 2.7 用户分级管理
#### 2.7.1 超级用户（admin）
初始账号：admin，初始密码：Zqlt_123456789
- 全量用户/APP/角色/服务管理权限；
- 审核服务注册、配置全局限流规则、调整健康检测全局参数；
- 查看所有日志、手动移除永久黑名单、全局安全配置；
- 不可删除自身账号、不可取消自身超级管理员权限。

#### 2.7.2 普通用户
- 仅管理自身账号、自建APP、自建角色、自建服务；
- 仅配置自有服务限流规则、查看自身关联日志；
- 无服务审核、无全局限流配置、无永久黑名单移除权限。

## 3. 架构设计
### 3.1 整体架构
采用Go语言分层架构，前后端一体化：
1. 前端UI层：静态资源内嵌Go二进制，轻量化模板实现，访问 `/ui/` 直达；
2. 路由接口层：RESTful业务API + OIDC标准协议路由 + UI静态路由；
3. 中间件层：JWT鉴权、权限校验、限流、跨域、请求日志；
4. 业务逻辑层：鉴权、M2M签名、角色权限、限流令牌桶、黑名单、服务注册发现、健康检测、OIDC协议处理；
5. 数据层：PostgreSQL持久化、Redis缓存/分布式限流/令牌桶/黑名单/服务列表缓存。

### 3.3 数据存储设计
#### 3.3.1 PostgreSQL核心表
用户表、APP表、角色表、权限表、角色权限关联表、用户角色关联表、APP角色关联表、服务表、日志表、限流规则表、黑名单表、限流统计表。
敏感字段：密码、APPSecret 均加密存储，不存明文。

#### 3.3.2 Redis缓存设计
统一Key规范：**所有 Redis Key 必须携带 `serviceCode` 一级前缀**，格式为 `{serviceCode}:{module}:{submodule}:{identifier}`。`serviceCode` 是服务全局唯一编码，只允许小写字母和数字，长度3-16位。

```
{serviceCode}:user:password:{userId}
{serviceCode}:jwt:access:{jti}
{serviceCode}:jwt:refresh:{jti}
{serviceCode}:jwt:blacklist:{jti}
{serviceCode}:limit:bucket:{serviceId}:{dimension}:{value}
{serviceCode}:limit:stats:{serviceId}:{yyyyMMddHHmm}
{serviceCode}:service:list
{serviceCode}:service:health:{serviceId}
{serviceCode}:blacklist:{type}:{key}
{serviceCode}:whitelist:{type}:{key}
{serviceCode}:auth:fail:{type}:{key}
{serviceCode}:auth:lock:user:{userId}
{serviceCode}:m2m:nonce:{appId}:{timestamp}:{signHash}
{serviceCode}:oidc:code:{code}
```
统一过期策略、缓存淘汰机制，适配鉴权、限流、黑名单、白名单、M2M防重放、OIDC授权码、服务健康状态缓存。多服务共享 Redis 时，必须同时通过服务前缀和 Redis ACL 隔离数据。

## 4. 约束与边界情况
### 4.1 数据约束
- 用户名：唯一非空，3-20位，仅字母、数字、下划线；
- 密码：8-20位，必须包含大小写字母、数字、特殊符号，bcrypt加密存储；
- APP ID：系统自动8位字母数字，唯一不可修改；
- APP Secret：16位字母数字特殊符号，仅创建时可见、加密存储；
- 服务地址：唯一合规URL，端口1-65535；
- 权限标识：格式`模块:操作`，全局唯一；
- 限流规则：同服务同维度同粒度仅一条规则，阈值为正整数；
- 黑名单：同主体同服务仅一条记录，永久黑名单不可重复新增。

### 4.2 业务边界
- 鉴权边界：支持用户名密码登录、JWT无感刷新与自动续签、M2M标准签名鉴权；Access Token 默认短有效期，Refresh Token 默认长有效期并支持轮换；M2M时间戳有效期30s；
- 权限边界：普通用户严格越权拦截；管理员不可自删账号；服务必须审核才可接入发现；
- 限流边界：仅管控网关入口接口，不限制下游内部调用；多维度组合取最严格阈值；Redis宕机自动降级；
- 服务管理边界：7天未审核服务自动删除；服务删除不级联清理历史数据；健康检测依赖健康接口；
- 黑名单边界：临时黑名单到期自动清零计数；永久黑名单仅管理员可手动解封；按服务独立计数不互通。

### 4.3 异常处理边界
- PostgreSQL宕机：只读缓存可用，无法新增/编辑配置类操作；
- Redis宕机：鉴权降级直连数据库，限流切本地固定阈值；
- 网络异常：下游服务标记异常，恢复后自动重试检测；
- 配置异常：限流阈值非法自动继承全局规则；
- 日志完整记录各类异常事件，支持问题溯源。

## 5. 部署与初始化
### 5.1 部署环境要求
- 操作系统：CentOS7+/Windows Server 2016+
- 依赖：PostgreSQL12+、Redis6.0+
- 运行：Go1.20+编译二进制，无需额外运行时
- 端口：默认8080，UI同端口 `/ui/` 访问

### 5.2 初始化步骤
1. 部署PostgreSQL并创建数据库 `auth_limit_db`；部署Redis并开启持久化；
2. 部署Go编译后的二进制服务，配置数据库、Redis、serviceCode、JWT、OIDC、限流、健康检测参数；
3. 服务启动自动执行数据库建表、初始化超级管理员admin、预设基础权限、初始化全局限流默认规则；
4. 直接访问 `http://IP:8080/ui/` 登录后台即可使用；
5. 无需单独部署前端，UI已内嵌一体化。

## 7. APISIX 集成规范
### 7.1 集成原则
本服务**原生标准OIDC协议**，直接使用APISIX官方 `openid-connect` 插件，**无需开发自定义Lua插件**，降低网关侵入与维护成本。

### 7.2 集成流程
1. APISIX 配置 openid-connect 插件，指向本服务OIDC发现地址；
2. 网关自动完成用户登录跳转、令牌获取、RSA公钥自动验签；
3. 网关截取请求用户/APP/IP/服务标识，调用本服务 `/oidc/limit/verify` 做限流二次校验；
4. 鉴权+限流全部通过则放行，失败分别返回401/403/429标准错误码。

### 7.3 插件配置
直接使用官方标准配置，仅需填写本服务OIDC发现地址、客户端ID与密钥即可。

## 8. 安全规范
- 密码bcrypt加密存储，废弃MD5；
- OIDC采用RSA非对称签名，防令牌篡改；
- JWT采用Access Token + Refresh Token，支持自动续签、刷新令牌轮换和登出吊销；
- APPSecret加密存储，严格控制可见范围；
- 生产环境强制HTTPS传输；
- 接口入参防注入、防XSS、防暴力破解、防请求重放；
- 黑名单全局拦截恶意主体访问；
- 定期数据库备份、Redis持久化，防止数据丢失。

## 9. 维护与监控
### 9.1 日常维护
- 数据库定时备份，避免数据丢失；
- 日志按周期归档清理，防止存储溢出；
- Redis缓存定期治理，清理无效缓存键；
- 限流规则根据业务流量变化定期调优；
- 清理长期无效的黑名单记录；
- 每日巡检服务运行状态（进程、端口、资源占用）。

### 9.2 监控指标

## 10. 验收清单

### 10.1 功能验收
- [ ] 用户可注册、登录、登出；
- [ ] 密码使用 bcrypt 存储，数据库不保存明文密码；
- [ ] 连续5次密码错误后账号锁定15分钟；
- [ ] 登录成功返回 Access Token 和 Refresh Token；
- [ ] Access Token 临近过期时自动续签，并通过响应头返回新 Token；
- [ ] Access Token 过期、Refresh Token 有效时可调用 `/api/auth/refresh` 无感续签；
- [ ] 登出后 Access Token 和 Refresh Token 均失效；
- [ ] M2M APP 可创建、重置 Secret，Secret 明文只出现一次；
- [ ] M2M HMAC-SHA256 签名校验正确，超时和重放请求被拒绝；
- [ ] OIDC discovery、authorize、token、jwks、userinfo 端点可用；
- [ ] APISIX 可通过 openid-connect 插件完成发现与 JWKS 验签；
- [ ] UI 可通过 `/ui/` 访问并完成核心管理操作；
- [ ] 普通用户只能管理自有资源；
- [ ] 超级管理员可管理全局资源，但不能删除自己或取消自身超级管理员权限；
- [ ] 服务注册、审核、发现、健康检测链路可用；
- [ ] Redis 令牌桶限流生效，超限返回 429；
- [ ] Redis 不可用时自动切换本地限流；
- [ ] 黑名单命中返回 403；
- [ ] 白名单主体可跳过限流；
- [ ] 日志和限流统计可按权限查询。

### 10.2 Redis隔离验收
- [ ] 所有 Redis Key 均以 `{serviceCode}:` 开头；
- [ ] 代码中不存在无前缀 Redis Key；
- [ ] Redis 客户端会拦截跨 `serviceCode` Key 访问；
- [ ] Lua 脚本和批量命令会校验所有 Key；
- [ ] Redis ACL 只允许当前服务用户访问 `{serviceCode}:*`；
- [ ] 缓存清理只清理当前 `serviceCode` 前缀。

### 10.3 安全验收
- [ ] JWT 使用 RS256 签名；
- [ ] 私钥不进入代码仓库；
- [ ] 生产环境强制 HTTPS；
- [ ] 接口参数完成基础校验和注入防护；
- [ ] 普通用户越权访问返回 403；
- [ ] APPSecret 不以明文落库或写日志；
- [ ] 黑名单、登录失败、M2M 失败均记录审计日志。

## 11. 测试用例

### 11.1 鉴权测试
| 用例 | 操作 | 预期 |
|---|---|---|
| 用户登录成功 | 调用 `/api/auth/login` | 返回 Access Token 和 Refresh Token |
| 密码错误计数 | 连续输入错误密码5次 | 账号锁定15分钟 |

| 登出失效 | 登出后再次访问鉴权接口 | 返回401 |

### 11.2 M2M测试
| 用例 | 操作 | 预期 |
|---|---|---|

| 时间戳超时 | timestamp 超过30秒偏差 | 返回401 |
| 重放请求 | 重复发送相同 timestamp 和 sign | 第二次返回401 |
