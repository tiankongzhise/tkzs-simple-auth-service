# 鉴权限流服务开发文档（完整平台对齐版）

## 0. 文档说明
本文档以《简单鉴权与限流服务产品文档（最终开发对齐版）》为范围基准，开发目标为完整平台，而不是仅实现 JWT 鉴权和 Redis 限流模块。

本版本统一以下口径：

1. 按产品文档实现完整平台：网页登录鉴权、JWT 自动续签、M2M 鉴权、OIDC/OAuth2、APISIX 联动、内嵌 UI、RBAC、服务注册发现、健康检测、限流、黑白名单、日志、监控与降级。
2. 所有 Redis Key 必须携带 `serviceCode` 一级前缀。
3. JWT 支持刷新，并实现无感刷新、自动续签。
4. 产品文档与开发文档的接口、数据模型、配置、验收标准保持一致。

---

## 1. 总体目标

### 1.1 平台能力
本服务提供轻量级统一鉴权、精细化限流和服务治理能力，面向接入 APISIX 网关的中小型服务集群。

核心能力包括：

1. 网页用户登录鉴权。
2. JWT 访问令牌、刷新令牌、无感刷新与自动续签。
3. M2M 机器鉴权，基于 APPID、APPSecret、HMAC-SHA256 签名。
4. OIDC/OAuth2 标准协议端点，兼容 APISIX 官方 `openid-connect` 插件。
5. 用户、APP、角色、权限、服务、限流规则、黑白名单、日志的内嵌管理 UI。
6. Redis 分布式令牌桶限流，本地令牌桶降级。
7. 服务注册、服务发现、健康检测、服务状态同步。
8. PostgreSQL 持久化，Redis 缓存与分布式状态存储。
9. 多服务共享 Redis 时，通过 `serviceCode` 前缀和 Redis ACL 实现隔离。

### 1.2 技术栈
| 类别 | 选型 |
|---|---|
| 后端语言 | Go 1.20+ |
| Web 框架 | Gin |
| ORM | GORM v2 |
| 数据库 | PostgreSQL 12+ |
| 缓存 | Redis 6.0+，开启 AOF + RDB |
| 配置 | Viper |
| JWT | RS256 签名，Access Token + Refresh Token |
| OIDC/OAuth2 | 标准发现、授权、Token、UserInfo、JWKS |
| M2M 签名 | HMAC-SHA256 |
| 密码哈希 | bcrypt |
| 限流 | Redis 分布式令牌桶 + 本地内存令牌桶降级 |
| 前端 | Go 内嵌静态资源或模板，HTMX 可选 |

启动时必须校验：

1. `service.code` 非空，匹配正则 `^[a-z0-9]{3,16}$`。
2. Redis 连接可用时，当前 Redis 用户 ACL 只允许访问 `{service.code}:*`。
3. JWT 私钥、公钥存在且可解析。
4. OIDC issuer 与 server host 在生产环境中必须使用 HTTPS。
5. 限流阈值、健康检测间隔、锁定时间必须为正数。

## 4. Redis Key 规范

### 4.1 强制规则
所有 Redis Key 必须携带 `serviceCode` 一级前缀，格式：

```text
{serviceCode}:{module}:{submodule}:{identifier}
```

示例：

```text
authlimit:jwt:access:{jti}
authlimit:limit:bucket:{serviceId}:{dimension}:{value}
```

业务代码禁止硬编码无前缀 Key，必须通过统一工具生成。

### 4.2 Key 清单
| 用途 | Key 格式 | TTL |
|---|---|---|
| 用户密码缓存 | `{serviceCode}:user:password:{userId}` | 30 分钟 |
| Access Token 状态 | `{serviceCode}:jwt:access:{jti}` | access token 剩余有效期 |
| Refresh Token 状态 | `{serviceCode}:jwt:refresh:{jti}` | refresh token 剩余有效期 |
| Token 黑名单 | `{serviceCode}:jwt:blacklist:{jti}` | token 剩余有效期 |
| 登录失败计数 | `{serviceCode}:auth:fail:{type}:{key}` | 10 分钟 |
| 账号锁定 | `{serviceCode}:auth:lock:user:{userId}` | 锁定时长 |
| M2M 防重放 | `{serviceCode}:m2m:nonce:{appId}:{timestamp}:{signHash}` | 60 秒 |
| 限流令牌桶 | `{serviceCode}:limit:bucket:{serviceId}:{dimension}:{value}` | 按规则 |
| 限流统计 | `{serviceCode}:limit:stats:{serviceId}:{yyyyMMddHHmm}` | 统计周期 |
| 白名单 | `{serviceCode}:whitelist:{type}:{key}` | 按配置 |
| 黑名单 | `{serviceCode}:blacklist:{type}:{key}` | 按配置 |
| 服务列表 | `{serviceCode}:service:list` | 常驻 |
| 服务健康 | `{serviceCode}:service:health:{serviceId}` | 健康检测周期 x 2 |
| OIDC 授权码 | `{serviceCode}:oidc:code:{code}` | 5 分钟 |

### 4.3 Redis 安全客户端
必须封装 Redis 客户端，所有读写命令在执行前校验 Key 前缀。

必须覆盖常用命令：

1. `Get`、`Set`、`Del`、`Exists`、`Expire`。
2. `Incr`、`IncrBy`、`HGet`、`HSet`、`SAdd`、`SMembers`。
3. `Eval`、`EvalSha` 必须校验传入的所有 `KEYS`。
4. 批量删除只允许扫描 `{serviceCode}:*`。

生产环境必须同时启用 Redis ACL：

```redis
ACL SETUSER authlimit_user ON >password123 ~authlimit:* +@all
ACL SETUSER default OFF
ACL SAVE
```

## 6. 鉴权设计

### 6.1 网页登录
接口：`POST /api/auth/login`

流程：

1. 校验用户名、密码格式。
2. 检查账号是否被锁定。
3. 优先读取 Redis 密码缓存，未命中查询 PostgreSQL。
4. 使用 bcrypt 校验密码。
5. 失败时更新登录失败计数，达到阈值写入账号锁定 Key。
6. 成功后清理失败计数，签发 Access Token 和 Refresh Token。
7. Token 元数据写入 PostgreSQL，Token 状态写入 Redis。

### 6.2 JWT 自动续签
本平台使用 Access Token + Refresh Token 双令牌机制。

规则：

1. Access Token 默认 30 分钟过期。
2. Refresh Token 默认 24 小时过期。
3. 客户端请求业务接口时携带 `Authorization: Bearer {accessToken}`。
4. 当 Access Token 剩余有效期小于 `jwt.auto_refresh_before_minutes` 时，服务端自动签发新 Access Token。
5. 新 Access Token 通过响应头返回：
   - `X-Access-Token`
   - `X-Access-Token-Expire-At`
6. 当 Access Token 已过期但 Refresh Token 仍有效时，客户端调用 `POST /api/auth/refresh` 无感刷新。
7. `refresh_rotate=true` 时，每次刷新同时轮换 Refresh Token，并废弃旧 Refresh Token。
8. 登出时 Access Token 和 Refresh Token 全部加入黑名单。

### 6.3 JWT Claims
Access Token 必须包含：

```json
{
  "iss": "authlimit",
  "sub": "user_id",
  "jti": "token_id",
  "typ": "access",
  "roles": ["admin"],
  "permissions": ["user:read"],
  "exp": 1710000000,
  "iat": 1709998200
}
```

Refresh Token 必须包含：

```json
{
  "iss": "authlimit",
  "sub": "user_id",
  "jti": "refresh_token_id",
  "typ": "refresh",
  "exp": 1710084600,
  "iat": 1709998200
}
```

### 6.4 M2M 鉴权
接口：`POST /api/auth/m2m`

请求头：

```text
appId: {appId}
timestamp: {unixSeconds}
sign: {hexHmacSha256}
```

签名规则：

1. 请求参数按 ASCII 升序排序。
2. 拼接格式：`APPSecret&timestamp&k1=v1&k2=v2`。
3. 使用 HMAC-SHA256 生成小写十六进制签名。
4. 服务端校验 APP 状态、时间戳误差不超过 30 秒、签名正确、nonce 未使用。
5. 验签成功后可签发机器访问 Token，或直接返回鉴权成功结果，具体由接口场景决定。

### 6.5 RBAC 权限隔离
权限判断必须同时满足：

1. 用户或 APP 拥有对应权限点。
2. 普通用户只能访问自己拥有的资源。
3. 超级管理员可访问全局资源，但不可删除自身账号、不可取消自身超级管理员权限。

资源隔离范围：

1. 用户。
2. APP。
3. 角色。
4. 服务。
5. 限流规则。
6. 日志。
7. 黑名单和白名单。

## 7. OIDC/OAuth2 设计

### 7.1 标准端点
| 功能 | 方法 | 路径 |
|---|---|---|
| OIDC 自动发现 | GET | `/.well-known/openid-configuration` |
| 授权端点 | GET | `/oauth2/authorize` |
| Token 端点 | POST | `/oauth2/token` |
| JWKS | GET | `/oauth2/jwks` |
| 用户信息 | GET | `/oauth2/userinfo` |

### 7.2 支持的 OAuth2 流程
首期必须支持：

1. Authorization Code Flow。
2. Refresh Token。
3. Client Credentials 可作为 M2M 与 APISIX 对接增强项。

### 7.3 APISIX 集成
APISIX 使用官方 `openid-connect` 插件接入：

1. `discovery` 指向 `/.well-known/openid-configuration`。
2. APISIX 通过 `/oauth2/jwks` 自动获取公钥验签。
3. 网关或业务入口调用 `/oidc/limit/verify` 完成限流校验。
4. 鉴权失败返回 401，权限不足返回 403，限流返回 429。

### 7.4 APISIX 限流校验接口
接口：`POST /oidc/limit/verify`

请求体示例：

```json
{
  "serviceId": "svc_001",
  "path": "/api/orders",
  "method": "GET",
  "ip": "127.0.0.1",
  "userId": "user_001",
  "appId": "app_001"
}
```

响应：

```json
{
  "allowed": true,
  "remaining": 10,
  "resetAt": 1710000000
}
```

## 8. 限流设计

### 8.1 算法
统一采用 Redis 分布式令牌桶算法。

要求：

1. 使用 Lua 脚本保证令牌读取、补充、扣减的原子性。
2. 支持 IP、用户、APP、接口多维度限流。
3. 多维度组合时，取最严格结果，任一维度超限即拒绝。
4. Redis 不可用时切换本地内存令牌桶。
5. Redis 恢复后自动恢复分布式限流。

### 8.2 限流规则
规则字段：

1. 服务 ID。
2. 维度：`ip`、`user_id`、`app_id`、`path`。
3. 粒度：秒、分钟、小时、天。
4. 容量。
5. 令牌生成速率。
6. 黑名单触发条件。
7. 封禁时长。
8. 是否启用。

### 8.3 响应规范
限流触发返回：

```text
HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1710000000
Retry-After: 10
```

## 9. 黑白名单

### 9.1 黑名单
黑名单类型：

1. IP。
2. 用户。
3. APP。
4. Token。

规则：

1. 临时黑名单默认封禁 15 分钟。
2. 临时黑名单累计触发达到阈值后可升级为永久黑名单。
3. 永久黑名单仅超级管理员可移除。
4. 黑名单请求返回 403。

### 9.2 白名单
白名单类型：

1. IP。
2. 用户。
3. APP。

白名单主体可跳过限流和黑名单限制，但仍需完成基础鉴权。

## 10. 服务治理

### 10.1 服务注册
接口注册和 UI 注册均支持。

规则：

1. 普通用户创建服务需要超级管理员审核。
2. 管理员创建服务直接生效。
3. 接口注册服务 7 天未审核自动删除。
4. 服务必须已审核且健康，才能进入服务发现列表。

### 10.2 服务发现
基于 Redis 缓存健康服务列表。

要求：

1. 服务上线、下线、审核状态、健康状态变化时同步 Redis。
2. 支持按服务名、健康状态查询。
3. 服务删除不级联删除历史日志和统计。

### 10.3 健康检测
规则：

1. 默认 30 秒检测一次。
2. 管理员可配置 10 秒到 5 分钟。
3. 默认路径 `/health`，支持服务级自定义。
4. 单次异常为亚健康。
5. 连续 3 次异常为异常，并移出服务发现列表。
6. 恢复后自动重新纳入服务发现列表。

## 12. 降级与异常处理

### 12.1 PostgreSQL 不可用
1. 登录、配置新增、配置编辑不可用。
2. 已缓存 Token 校验可继续。
3. UI 显示只读或错误提示。

### 12.2 Redis 不可用
1. 鉴权降级为直连 PostgreSQL。
2. 限流切换本地内存令牌桶。
3. 服务发现返回最近一次内存快照。
4. Redis 恢复后自动重建关键缓存。

### 12.3 配置异常
1. 非法限流规则自动继承全局默认规则。
2. 非法健康检测配置回退默认值。
3. 启动关键配置缺失时直接失败，不允许带病启动。

### 13.2 监控指标

3. Token 刷新次数、失败次数。
4. 限流次数、限流命中率。
5. Redis/PG 连接与读写耗时。
6. 下游服务健康率。

## 14. 验收清单

### 14.1 功能验收
- [ ] 用户可注册、登录、登出。
- [ ] 密码使用 bcrypt 存储，数据库无明文密码。
- [ ] 连续 5 次密码错误后账号锁定 15 分钟。
- [ ] 登录成功返回 Access Token 和 Refresh Token。
- [ ] Access Token 临近过期时自动续签，并通过响应头返回新 Token。
- [ ] Access Token 过期、Refresh Token 有效时可调用刷新接口无感续签。
- [ ] 登出后 Access Token 和 Refresh Token 均失效。
- [ ] M2M APP 可创建、重置 Secret，Secret 明文只出现一次。
- [ ] M2M HMAC-SHA256 签名校验正确。
- [ ] M2M 时间戳超过 30 秒被拒绝。
- [ ] 重放请求被拒绝。
- [ ] OIDC discovery、authorize、token、jwks、userinfo 端点可用。
- [ ] APISIX 可通过 openid-connect 插件完成发现与 JWKS 验签。
- [ ] UI 可通过 `/ui/` 访问并完成核心管理操作。
- [ ] 普通用户只能管理自有资源。
- [ ] 超级管理员可管理全局资源，但不能删除自己或取消自身超级管理员权限。
- [ ] 服务注册、审核、发现、健康检测链路可用。
- [ ] Redis 令牌桶限流生效，超限返回 429。
- [ ] Redis 不可用时自动切换本地限流。
- [ ] 黑名单命中返回 403。
- [ ] 白名单主体可跳过限流。
- [ ] 日志和限流统计可按权限查询。

### 14.2 Redis 隔离验收
- [ ] 所有 Redis Key 均以 `{serviceCode}:` 开头。
- [ ] 代码中不存在无前缀 Redis Key。
- [ ] `SafeRedisClient` 拦截跨 `serviceCode` Key 访问。
- [ ] `Eval` 和批量命令会校验所有 Key。
- [ ] Redis ACL 只允许当前服务用户访问 `{serviceCode}:*`。
- [ ] 清理缓存时只清理当前 `serviceCode` 前缀。

### 14.3 安全验收
- [ ] JWT 使用 RS256 签名。
- [ ] 私钥不进入代码仓库。
- [ ] 生产环境强制 HTTPS。
- [ ] 接口参数完成基础校验和注入防护。
- [ ] 普通用户越权访问返回 403。
- [ ] APPSecret 不以明文落库或写日志。
- [ ] 黑名单、登录失败、M2M 失败均记录审计日志。

## 15. 测试用例

### 15.1 鉴权测试
| 用例 | 前置条件 | 操作 | 预期 |
|---|---|---|---|
| 用户登录成功 | 用户存在且密码正确 | 调用 `/api/auth/login` | 返回 access token 和 refresh token |
| 密码错误计数 | 用户存在 | 连续输入错误密码 5 次 | 账号锁定 15 分钟 |
| 自动续签 | Access Token 剩余有效期小于阈值 | 调用任意鉴权接口 | 响应头返回新 Access Token |
| 刷新成功 | Access Token 过期，Refresh Token 有效 | 调用 `/api/auth/refresh` | 返回新令牌 |
| 刷新失败 | Refresh Token 过期或黑名单 | 调用 `/api/auth/refresh` | 返回 401 |
| 登出失效 | 用户已登录 | 调用 `/api/auth/logout` 后再次访问 | 返回 401 |

### 15.2 M2M 测试
| 用例 | 操作 | 预期 |
|---|---|---|
| 正确签名 | 使用正确 APPSecret 计算 sign | 返回成功 |
| 错误签名 | 修改任一参数后复用 sign | 返回 401 |
| 时间戳超时 | timestamp 超过允许偏差 | 返回 401 |
| 重放请求 | 重复发送相同 timestamp 和 sign | 第二次返回 401 |
| Secret 重置 | 重置 APPSecret 后使用旧 Secret | 返回 401 |

### 15.3 OIDC/APISIX 测试
| 用例 | 操作 | 预期 |
|---|---|---|
| Discovery | GET `/.well-known/openid-configuration` | 返回 issuer、jwks_uri 等字段 |
| JWKS | GET `/oauth2/jwks` | 返回可验签公钥 |
| 授权码流程 | 完成 authorize 和 token 交换 | 返回 token |
| UserInfo | 携带有效 token 调用 `/oauth2/userinfo` | 返回用户信息 |
| APISIX 验签 | APISIX 使用 JWKS 验证 token | 验签成功 |

### 15.4 权限测试
| 用例 | 操作 | 预期 |
|---|---|---|
| 普通用户访问他人 APP | 查询或编辑他人 APP | 返回 403 |
| 普通用户越权赋权 | 分配自己没有的权限 | 返回 403 |
| 管理员管理全局资源 | 编辑任意服务或规则 | 成功 |
| 管理员删除自己 | 删除自身账号 | 返回 400 或 403 |

### 15.5 限流测试
| 用例 | 操作 | 预期 |
|---|---|---|
| 未超限 | 低于阈值请求 | 正常放行 |
| 超限 | 超过令牌桶阈值 | 返回 429 |
| 多维度取严 | IP 未超限但用户超限 | 返回 429 |
| Redis 降级 | 停止 Redis 后请求 | 使用本地令牌桶 |
| Redis 恢复 | 恢复 Redis | 自动切回 Redis 限流 |

### 15.6 Redis 隔离测试
| 用例 | 操作 | 预期 |
|---|---|---|
| 标准 Key | 写入业务缓存 | Key 以 `{serviceCode}:` 开头 |
| 跨服务读取 | 读取其他前缀 Key | 被客户端拒绝 |
| Lua 跨 Key | Eval 传入其他前缀 Key | 被客户端拒绝 |
| ACL 校验 | 使用 Redis 用户访问其他前缀 | Redis 拒绝 |
